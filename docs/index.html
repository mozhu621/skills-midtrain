Skill Spec Midtraining

让模型在用工具之前，
先读懂每个工具的使用边界

把 Anthropic 的 Model Spec Midtraining 配方迁移到 skill / tool 域： 为每个 skill 合成一份五要素「使用规范」，再写成多体裁讨论语料 midtrain 基座模型—— 在做工具调用 SFT 之前，就内化「何时该调、何时不该调」的判断。

核心假设：先用自然语言把使用边界讲透，能让对齐训练更好地泛化到训练分布之外的新情况， 而不只是「看到这种请求就发这个调用」的模式匹配。

GitHub 仓库 → MSM 论文 (arXiv:2605.02087)

这是在解决什么问题

直接在工具调用轨迹上做 SFT，模型容易学成表层的模式匹配。SSM 主张：把判断依据写成规范、教给模型。

仅 SFT 轨迹

学到「条件反射」

从「请求 → 调用」的配对里，模型记住的是表面触发模式。遇到该澄清、该改用别的 skill、或根本不该调用的新场景时，往往照旧硬调。

SSM 先 midtrain

学到「判断依据」

先用大量自然语言文档把每个 skill 的能力边界、禁用场景、易混对象、失败方式与验证方法讲透，再接 SFT——判断更能迁移到没见过的情况。

五要素 Skill Spec

每个 skill 被结构化成五个要素，每个要素含一段说明文字 + 2–5 条可独立引用的原子原则（principles）。

S1

能力与边界

做什么、输入/输出契约，并明确说明不能做什么。严格来自 skill 文档。

S2

何时不该用

能用自身知识或更简单手段解决、前置条件未满足（应澄清）、超出范围（应直说）。

S3

对比辨析

与库内相似 skill 怎么二选一，给可迁移的判据。相似项由 TF-IDF 近邻自动注入。

S4

失败方式

幻觉参数、违反顺序/状态假设、误读输出、部分失败——每条带迷你案例与根因。

S5

调用后验证

用完怎么自检结果，以及重试 / 换方案 / 如实报告失败的策略。

合成数据流水线

六个阶段，从原始 skill 库一路到可训练的 token 语料。

01

采集

克隆 skill 仓库、收割市场站点、拉取工具池（Seal-Tools / xLAM）。

configs/sources.yaml → data/raw/
02

解析入库

统一成一份 schema，去重。

→ data/skills.jsonl · 16,983 skills
03

Stage A · 生成 Spec

skill → 五要素 Skill Spec（带 S3 近邻注入、字段校验）。

gen_specs.py → data/specs/specs.jsonl
04

Stage B · 生成文档

两段式（点子 → 成文）多体裁讨论语料，过泄漏/去重质量闸。

gen_docs.py + quality.py → docs_clean.jsonl
05

Stage C · 打包

tokenize、文档间插 EOS、切 4096 块、uint32 memmap，可混回放语料。

build_corpus.py → data/corpus/*.bin
06

Stage C · Midtrain

Qwen3.5-4B-Base 低强度续训（lr 2e-5，512k tok/step，8×GPU）。

train_midtrain.py → checkpoints/

关键设计

几条让语料「干净、有信息量、不泄漏」的核心约束。

防 SFT 格式泄漏

midtrain 语料严禁出现工具调用轨迹格式（<tool_call>、"arguments":、Action:/Observation: 等）——提示词端禁止 + 正则质量闸双保险。

防术语回声

写文档时喂给模型的是无标签摘要（不出现 S1–S5 字样），目标是用自然语言多样重述，而非复读规范条文。

Assertion-targeting

每篇文档只聚焦少数要素的原则，整套语料在 skill × 体裁 × 要素上交叉覆盖——单篇有焦点，语料有广度。

防虚构

禁止编造日期/作者/引用/链接，禁止把臆造的工具输出或「测得的数字」当真实事实写；审计体裁只能用定性量级。

16 种体裁

教程、FAQ、Slack 辩论、事故复盘、审计报告……带权重采样；正式/非正式体裁对「规范」的引用口径不同。

接地的 Spec

S1/S4 关于行为与参数的断言必须来自 skill 文档，不得发明能力；S2/S3/S5 允许通用工程判断补全。
